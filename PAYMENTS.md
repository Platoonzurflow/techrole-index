# Платежи TechRole Index

Актуальность сравнения: 22 июля 2026 года. Это техническая памятка, а не юридическая или налоговая консультация. Публичные тарифы могут измениться; договор и тариф в личном кабинете провайдера имеют приоритет. Владелец подтвердил статус самозанятого на НПД.

## Принятое техническое решение

Основной кандидат для подтверждённого статуса НПД — **Robokassa**, резервный — **ЮKassa**.

Robokassa стала основной после подтверждения НПД: её официальная страница прямо поддерживает самозанятых без ИП и бесплатную автоматическую отправку чеков через «Робочеки СМЗ». ЮKassa технически зрелее и на публичном тарифе дешевле для части карточных платежей, но с 29 декабря 2025 года не автоматизирует чеки самозанятых; их пришлось бы вручную создавать в «Мой налог».

Текущий сайт использует только локальный `demo` sandbox. Реализованы два внешних адаптера: ЮKassa и Robokassa. Robokassa формирует подписанный server-side redirect, подписывает номенклатуру чека, проверяет ResultURL по Паролю №2, отвечает обязательным `OK<InvId>`, дедуплицирует повторы, поддерживает live refund API по Password3 и сверяет его состояние. Официальный test shop владельца ещё не вызывался: нет выданных владельцем тестовых параметров. Production-списания заблокированы конфигурационными guard-проверками.

## Что владелец обязан сделать лично

1. Страна получателя — Россия, статус — самозанятый НПД. До live остаётся проверить, что продаваемая услуга разрешена НПД, годовой доход не превышает действующий лимит и нет запрещённых посреднических/агентских отношений.
2. Самостоятельно зарегистрировать этот статус и открыть личный кабинет провайдера. Самозанятому отдельный расчётный счёт по НПД сам по себе не обязателен; ИП/компании обычно указывают расчётный счёт для выплат. Конкретное требование проверяется в договоре провайдера.
3. Пройти KYC и заключить договор: предоставить ИНН, паспорт и селфи/идентификацию; для ИП — ОГРНИП/данные ЕГРИП и при необходимости лицензии/доверенность; для компании — ОГРН, уставные и банковские реквизиты, сведения о руководителе, бенефициарах и полномочиях. Точный список формирует провайдер.
4. Для основного варианта подключить в Robokassa «Робочеки СМЗ» и проверить привязку «Мой налог». Сайт передаёт только server-side позицию `service`, полную оплату и `tax=none`; правильность предмета расчёта и наименования нужно подтвердить до live. При резервной ЮKassa чек создаётся вручную в «Мой налог» и передаётся покупателю. Возврат также должен быть отражён в НПД.
5. Заполнить реквизиты продавца, продукт, цену, правила оказания услуги, возвраты, сроки хранения данных и контакты в шаблонах `/legal/*`; перед live передать их юристу. Проверить уведомление Роскомнадзора, локализацию персональных данных и договоры с хостингом, провайдером, кассой/ОФД и почтой.
6. Получить постоянный HTTPS-домен и круглосуточный hosting. Tailscale Funnel подходит для теста, но зависит от включённого ПК; при выключении webhook не будет принят вовремя.
7. В личном кабинете Robokassa создать магазин и тестовые Пароль №1/№2, выбрать тот же алгоритм подписи, что в `.env`, и настроить ResultURL. Для live отдельно создать Password3 для возвратов. Не присылать пароль от кабинета, SMS-коды, данные Госуслуг, полный номер карты или CVC.
8. Указать ResultURL `https://<постоянный-домен>/api/v1/payments/webhooks/robokassa` методом POST и проверить ответ `OK<InvId>`. Success/Fail URL передаются сервером в каждом заказе. Для резервной ЮKassa webhook остаётся `https://<домен>/api/v1/payments/webhooks/yookassa`.
9. Явно подтвердить включение production-списаний после полного теста. Без этого `PAYMENTS_MODE=live` не включается.

Нельзя выполнить вместо владельца: зарегистрировать НПД/ИП/компанию, открыть банковский счёт, пройти паспортный KYC/селфи, подписать договор или оферту, подтвердить SMS/КЭП/Госуслуги, выбрать налоговый режим и ставку НДС, зарегистрировать кассу в ФНС, дать юридическое заключение, принять договор провайдера и дать согласие на реальные списания.

## Что уже реализовано

- серверный каталог одного продукта `premium_30_days`; браузер не передаёт цену;
- `payment_orders`, `payment_refunds`, версии принятой оферты и аудит sanitized payment events;
- client/provider idempotency и защита от повторных webhook;
- строгая сверка order id, payment id, суммы, валюты и test/live режима;
- HMAC-аутентификация локального demo webhook;
- для ЮKassa — Basic auth к официальному API, `Idempotence-Key` и повторный авторизованный GET для проверки webhook вместо доверия присланному JSON;
- для Robokassa — уникальный числовой `InvId`, подписи Паролем №1/№2, allowlisted hash algorithm, server-side `Shp_order_id`, проверка суммы/режима, form-urlencoded ResultURL и обязательный plain-text ack;
- server-side Robokassa Receipt для услуги НПД; карточные данные и произвольная номенклатура браузера не принимаются;
- live refund API Robokassa подписывается HS256/Password3, получает OpKey только через подписанный OpStateExt и сверяется фоновой задачей каждые пять минут; официальный OpStateExt не поддерживает тестовые операции, поэтому refund-сценарий без денег проверяется в demo и mock-контракте;
- только серверное предоставление Premium после `succeeded`, терминальные отмены и отзыв конкретного entitlement после полного возврата;
- server-side receipt item для ЮKassa из продукта и email пользователя, когда выбран режим кассы; ставка НДС задаётся только после юридического подтверждения;
- страницы sandbox checkout, ожидания, успеха и ошибки; статус успеха читается с сервера;
- шаблоны оферты, возвратов, политики данных и согласий с явными незаполненными полями;
- backend, provider, component и Playwright тесты, включая неверную подпись, подмену суммы, повтор, отмену и возврат.

Карточные реквизиты не принимаются и не хранятся приложением. Для реального сценария пользователь переходит на HTTPS-страницу провайдера.

## Локальный sandbox

В `.env`:

```dotenv
PAYMENTS_ENABLED=true
PAYMENTS_PROVIDER=demo
PAYMENTS_MODE=test
PAYMENTS_LIVE_CONFIRMED=false
PAYMENTS_LEGAL_APPROVED=false
PAYMENTS_STABLE_HTTPS_CONFIRMED=false
PAYMENTS_TERMS_VERSION=draft-2026-07-22
PAYMENTS_SELLER_STATUS=self_employed
PAYMENTS_FISCALIZATION_MODE=disabled
PREMIUM_30_DAYS_PRICE_RUB=1.00
```

Затем:

```powershell
docker compose run --rm migrate
docker compose up -d backend frontend
```

Зарегистрируйте новый аккаунт, откройте `/account`, примите тестовые условия и нажмите «Тестовая оплата». На локальной странице нет полей карты; кнопки только имитируют успешный или отменённый результат.

## Тестовый магазин Robokassa — основной путь

Владелец создаёт test shop и локально добавляет:

```dotenv
PAYMENTS_ENABLED=true
PAYMENTS_PROVIDER=robokassa
PAYMENTS_MODE=test
PAYMENTS_LIVE_CONFIRMED=false
PAYMENTS_LEGAL_APPROVED=false
PAYMENTS_SELLER_STATUS=self_employed
PAYMENTS_FISCALIZATION_MODE=disabled
ROBOKASSA_MERCHANT_LOGIN=<test MerchantLogin>
ROBOKASSA_PASSWORD1=<test Password #1>
ROBOKASSA_PASSWORD2=<test Password #2>
ROBOKASSA_HASH_ALGORITHM=sha256
```

Алгоритм в технических настройках магазина обязан совпадать с `ROBOKASSA_HASH_ALGORITHM`. Для проверки автоматического чека после подключения тестовой фискализации заменить mode на `robokassa`; номенклатуру и сумму всё равно формирует сервер. Password3 в тестовом контуре не нужен: официальный OpStateExt работает только в основном режиме.

## Тестовый магазин ЮKassa — резерв

Владелец создаёт test shop и локально добавляет:

```dotenv
PAYMENTS_ENABLED=true
PAYMENTS_PROVIDER=yookassa
PAYMENTS_MODE=test
PAYMENTS_LIVE_CONFIRMED=false
PAYMENTS_LEGAL_APPROVED=false
YOOKASSA_SHOP_ID=<test shopId>
YOOKASSA_SECRET_KEY=<test secret key>
```

Для sandbox можно оставить `PAYMENTS_FISCALIZATION_MODE=disabled`, если тестовый магазин не подключён к тестовой кассе. Если проверяются чеки, выбрать договорной режим и тестовый `YOOKASSA_VAT_CODE` после подтверждения бухгалтера. Секреты остаются только в `.env`/secret store, никогда в Git, issue, логе или browser bundle.

## Guard для production

Live-конфигурация намеренно не стартует, пока одновременно не выполнены все условия:

- `PAYMENTS_ENABLED=true`, `PAYMENTS_PROVIDER=robokassa`, `PAYMENTS_MODE=live`;
- `PAYMENTS_STABLE_HTTPS_CONFIRMED=true` только после проверки постоянного домена, TLS и круглосуточного host;
- production `ROBOKASSA_MERCHANT_LOGIN`, Password №1/№2 и Password3 только в secret store;
- `PAYMENTS_LIVE_CONFIRMED=true` после явного решения владельца;
- `PAYMENTS_LEGAL_APPROVED=true` и версия оферты без префикса `draft-`;
- подтверждён `PAYMENTS_SELLER_STATUS`;
- для НПД выбран `PAYMENTS_FISCALIZATION_MODE=robokassa` и в кабинете активированы «Робочеки СМЗ»;
- `ROBOKASSA_HASH_ALGORITHM` совпадает с техническими настройками магазина;
- все Robokassa payment/refund URL остаются официальными значениями из `.env.example`;
- `PUBLIC_BASE_URL` и `FRONTEND_ORIGIN` указывают на один постоянный HTTPS origin.

После переключения выполнить тест минимальной суммы разрешённой провайдером, проверить платёж, чек, повтор webhook, отмену и полный возврат в кабинете провайдера и локальной БД. Только затем разрешать обычную цену.

Текущий реализованный продукт — разовый доступ Premium на 30 дней в рублях, без автопродления. Перед test shop владелец должен подтвердить конечную цену и что не требуется подписка; изменить это решение можно отдельно, не доверяя параметрам браузера.

## Проверка готовности без раскрытия секретов

После входа администратором откройте `/admin`. Раздел «Готовность Robokassa» показывает два независимых checklist: тестовый магазин и реальные платежи. ResultURL можно скопировать в технические настройки магазина.

`GET /api/v1/admin/payment-readiness` возвращает только названия проверок и `ready: true/false`. MerchantLogin, Пароли №1/№2/№3 и ключи других провайдеров не сериализуются. Endpoint закрыт admin-auth; live остаётся заблокированным, пока не выполнены все legal, fiscalization, hosting и owner-confirmation guards.
